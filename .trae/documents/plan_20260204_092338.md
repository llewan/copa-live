# Implementación de Sincronización Bajo Demanda (On-Demand Sync)

## Problema
Vercel Free Tier limita los Cron Jobs a 1 por día. No podemos usar el cron de "10 minutos" para actualizar resultados en vivo.

## Solución Propuesta
Mover la lógica de actualización ("Smart Polling") al endpoint `GET /matches`. Cuando un cliente consulta los partidos, el servidor decidirá si es necesario actualizar desde la API externa, respetando un límite estricto de 15 minutos para no exceder la cuota diaria (100 reqs).

## Pasos de Implementación

### 1. Configuración de Vercel (`vercel.json`)
- Eliminar la tarea cron `live` (`*/10 * * * *`).
- Mantener la tarea cron `daily` (`0 6 * * *`) para la carga inicial del calendario.

### 2. Servicio de Auditoría (`auditService.ts`)
- Añadir método `getLastSyncTime(action: string): Promise<Date | null>`.
- Esto permitirá consultar cuándo fue la última vez que se ejecutó `syncLiveMatches` exitosamente.

### 3. Servicio de Fútbol (`footballService.ts`)
- Modificar `getMatches(date)`:
    1. Cargar partidos de la DB.
    2. Si `date` es HOY:
        - Verificar si hay partidos activos (`IN_PLAY`) o por comenzar.
        - Consultar `auditService.getLastSyncTime('syncLiveMatches')`.
        - Si (Ahora - ÚltimaSync) > **15 minutos**:
            - Ejecutar `syncLiveMatches()`.
            - Recargar partidos de la DB.
        - Si no, devolver datos cacheados (protección de cuota).

### 4. Cliente
- Confirmar que el cliente ya hace polling cada 5 minutos (ya implementado en `Dashboard` y `MatchDetail`).
- El cliente no necesita cambios, solo seguirá llamando a la API, y la API se encargará de darle datos frescos si corresponde.

## Beneficio
- **Ahorro de Cuota**: Solo se consume API si hay usuarios activos viendo la página.
- **Tiempo Real**: Los usuarios obtienen actualizaciones cada ~15 min durante los partidos.
- **Cero Costo en Vacío**: Si nadie visita la web, no se hacen llamadas a la API (a diferencia del Cron que corre siempre).
