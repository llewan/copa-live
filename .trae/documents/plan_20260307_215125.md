### Answer: What happens during live events?

When there are live matches, the system behaves as follows:

1.  **Frontend (Client)**:
    *   The dashboard automatically polls (requests data) every **5 minutes**.
    *   If you have 100 users, your server receives **100 requests every 5 minutes**.

2.  **Backend (Server)**:
    *   The server checks its cache (database).
    *   It only triggers a **fresh fetch** from the external API (API-Football) if the data is older than **15 minutes**.

### The Result:
*   **Efficiency**: Extremely high. No matter how many users you have, you only make **4 requests per hour** to the paid external API.
*   **Latency**: Users will see score updates with a delay of up to **15 minutes**.
    *   *Note: The frontend polls every 5 mins, but the backend only updates every 15 mins. This means 2 out of 3 polls will return the same cached data.*

### Proposed Plan

To improve this, I recommend two steps:

1.  **Implement Request Deduplication (Safety)**:
    *   **Goal**: Prevent race conditions where two users hitting the server at the exact same moment might trigger double API calls or get stale data.
    *   **Action**: Use a shared "Promise" in memory so all concurrent users wait for the same single update.

2.  **Align Intervals (Freshness)**:
    *   **Goal**: Make the "Live" experience feel faster.
    *   **Action**: Lower the backend sync interval from **15 minutes** to **5 minutes** to match the frontend.
    *   **Cost Warning**: This increases external API usage from ~100/day to ~300/day. *Only do this if your API plan allows >100 requests/day.*
